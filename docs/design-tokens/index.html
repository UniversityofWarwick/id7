Design tokens are an increasingly popular way to describe the various values and measurements that make up a standard design. They can be colours, sizes, fonts, and other values.

In ID7 we use a set of tokens in the form of LESS variables, which can be adjusted for individual applications and sites. However they have a few issues in practice:

  • There is no separation between "internal" and "public" variables, so it's difficult to know which variables you can reliably use and modify.
  • LESS variables are not dynamic, which limits how they can be used to theme and customise components.
  • Values tend to be offered individually, but in many cases sets of values go together well. A good example is pairs of colours for use as background and text colour - it's hard to know ahead of time if they will display with enough contrast to be readable.

With this in mind we will gradually be introducing design tokens implemented as CSS custom properties. We will clearly document which tokens can be safely used in your own designs, allowing you to implement interfaces that respond dynamically to theme changes and colour contrast, and maintain a consistent look and feel. Google's Material Design site has a good overview of the concept.

We haven't made any tokens public yet but the first ones will likely be colour pairings to be used together as background and foreground colours, which will have guaranteed contrast and automatically re-theme themselves when used across different sites.

Token naming system

Token names will have three parts:
  • The brand identifier - in this case w for Warwick
  • The token class:
    • ref - a reference token usually has a fixed value and wouldn't be used directly
    • sys - a system token is semantic and describes a particular use for a value. It is likely to be based on a reference token but will vary depending
  • The token description - explaining what it is or what it's for

An example (nonexistant) token name might be --w-sys-foregroundColor.
